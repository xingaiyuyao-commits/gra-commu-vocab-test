# 単語テスト週次サイクル 設計ドキュメント

**Goal:** 1グループが週1回集まって遊ぶ運用に合わせ、「週1〜3は単語テスト＋補助ゲーム1つ、週4は3週分の復習テスト」というサイクルを、サイトが状態管理・案内する仕組みを追加する。

**Scope:** このドキュメントは以下のサブプロジェクトのうち **サブプロジェクト1（週次サイクル管理）のみ** を対象とする。

1. ✅ 週次サイクル管理・復習テスト（本ドキュメントの対象）
2. ⬜ 週4の「3週分の単語を混ぜたゲーム」（コインタワー/並べ替えバトルへの単語データ接続。別途設計する）

サブプロジェクト2は、コインタワー・並べ替えバトルが単語テストとは無関係の固定問題プール（`questions.js` / `sentences.js`）を使っている現状があり、本サブプロジェクトより設計・実装コストが大きいため意図的に別スコープとした。

**運用ポリシー（ユーザー確定事項）:**

- 月4週サイクル。週1〜3は「単語テスト（1シリーズ）」を必ず実施し、その後に任意で🪙コインタワー/🧩並べ替えバトル/🔥Surviveのどれか1つを追加で行う
- 週4は「単語テスト」のみで、週1〜3で使った3シリーズ分の単語をまとめた復習テストを行う
- デプロイ先: Railway（Socket.IO常駐サーバーのため。Vercelはサーバーレスで常時接続・ファイル永続化と相性が悪く不採用）

## 前提となる既存アーキテクチャ

- `server.js`: Express + Socket.IO。コインタワー(`coin:*`)・並べ替えバトル(`order:*`)・単語テスト(`quiz:*`)のイベント群を持ち、各ルームの状態はメモリ上（`quizRooms`等）
- `wordtests.js` → `wordtests-clacel.js` / `wordtests-ielts.js` / `wordtests-toeic.js`: カテゴリ別・シリーズ別の問題データ
- `highscore.json`: コインタワーの最高記録をファイル永続化する既存パターン（`fs.readFileSync`/`writeFileSync`）
- 状態はすべて「サイト全体で1グループが使う」前提（マルチテナントではない）

## データモデル・永続化

`highscore.json` と同じパターンで `quiz-cycle.json` を新設する。

```json
{ "count": 0, "history": [] }
```

- `count`: ホストが「今週分を完了する」を押した累積回数
- `history`: 現サイクル（週1〜3）で実施したシリーズの記録。最大3件
  ```json
  [{ "category": "clacel", "seriesIndex": 0, "label": "Clacel 2.0 Series 1" }]
  ```
- **週番号**: `week = (count % 4) + 1`（count=0→第1週 … count=3→第4週 → count=4→第1週 で新サイクル）
- 第4週の完了時（`count % 4 === 3` の状態で完了を押した時）、`history` をクリアして次サイクルへ
- サーバー起動時に読み込み・保存の都度書き込み。ファイルが存在しない場合は `{ count: 0, history: [] }` から開始
- 状態はルームに紐づかない、サイト全体で共有される単一の値

## サーバー側API/イベント追加

### `GET /api/quiz-cycle`

トップページ用。Socket接続なしで週番号・履歴を取得する単純なHTTPエンドポイント。

```json
{ "week": 2, "history": [{ "category": "clacel", "seriesIndex": 0, "label": "Clacel 2.0 Series 1" }] }
```

### `quiz:completeSession`（ホストのみ・単語テストの結果画面から送信）

- 前提: `quiz:startGame`時にルーム状態へ `lastCategory` / `lastSeriesIndex` / `lastLabel` を保存しておく（現状は`room.questions`しか保持していないため追加が必要）。「完了」時はこの値を使う
- 呼び出し元のルームが今回実施したカテゴリ/シリーズを `history` に追記し、`count` を+1
- 第4週（復習）の完了時は `history` をクリアして次サイクルへ
- 処理後 `quiz-cycle.json` に保存し、呼び出し元に新しい `{ week, history }` を返す
- 非ホストからの呼び出し、`phase !== "finished"` 時の呼び出しは無視する（既存の`quiz:playAgain`と同じガード方式）

### `quiz:startReview`（週4のみ・ホストのみ）

- `history` の最大3シリーズから問題をプールし、`REVIEW_QUESTION_COUNT`（デフォルト20、コード上の定数1箇所）を上限に抽出
- 抽出アルゴリズム: 3シリーズに均等に割当てる（`quota = floor(N / 件数)`、余りは先頭シリーズから+1）。プールがシリーズ内に不足する場合はそのシリーズの全件を使い、余剰分を他シリーズへ回す。プール総数が`REVIEW_QUESTION_COUNT`未満ならプール全部を使う
- 見つからない（データファイルから削除された等の）シリーズは無視して残りだけでプールを作る。全シリーズが見つからない場合はエラーを返す
- 以降は既存の `quiz:started` と同じ形式でクライアントへ配信（クライアント側の出題・採点・結果画面はそのまま再利用できる）

## クライアント側UI

### トップページ（`index.html`）

ページ読み込み時に `GET /api/quiz-cycle` を叩き、単語テストカードの上にバナーを表示する。

- 週1〜3: 「📅 第◯週：単語テスト＋任意で🪙🧩🔥のどれか1つ」
- 週4: 「📅 第4週（復習週）：3週分の単語まとめテスト」

**ソフトガイドのみ**（表示するだけで、他ゲームのボタンを無効化・制限しない）。

### 単語テストのロビー画面（`quiz.html`）

- 週1〜3: 現行通りカテゴリ/シリーズのプルダウン＋「テスト開始」ボタン
- 週4: プルダウンを表示せず、「復習テストを開始」ボタンのみを表示（クリックで `quiz:startReview` を送信）

### 単語テストの結果画面

「もう一度」の隣に、ホストにのみ「今週分を完了する（次の週へ）」ボタンを追加。クリックで `quiz:completeSession` を送信し、返ってきた新しい週番号でその場の表示を更新する（トップページのバナーは次回読み込み時に反映される）。

## エッジケース

| ケース | 挙動 |
|---|---|
| 第4週到達時に`history`が3件揃っていない（ファイル破損・手動編集等） | 揃っている分だけでプールを作る。0件ならエラーメッセージを返す |
| 同じ週に単語テストを複数回やり直した | 「完了」ボタン押下時点でルームに設定されているシリーズのみを記録する（やり直し前の分は記録しない） |
| 同じ日に別ルームを2つ作り両方で「完了」を押した | 週が2つ進む。システムでは防止しない（少人数グループの運用に委ねる） |
| `history`のシリーズが後でデータファイルから削除されていた | 見つからないシリーズは無視して残りでプールを作る（クラッシュさせない） |
| 参加者（ホスト以外）の画面 | 週1〜4とも変更なし（「ホストの開始を待っています」のみ） |

## 明示的に対象外（サブプロジェクト2で扱う）

- 週4の「3週分の単語を混ぜたゲーム」（コインタワー/並べ替えバトルへの単語データ接続）
- Survive（外部Railwayリンクのまま。ローカル実装なし）
